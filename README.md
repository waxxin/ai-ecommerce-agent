# AI电商内容生成Agent系统 🤖

> 基于 LLM + RAG + Agent 的企业级电商内容自动生成平台

**一句话定义**：基于大语言模型构建的电商内容生成系统，实现商品标题、卖点与多平台文案的智能化自动生成。

---

## 📊 业务背景

### 电商内容生产的痛点

| 痛点             | 描述                               | 影响                         |
| ---------------- | ---------------------------------- | ---------------------------- |
| **人工成本高**   | 每条商品需人工撰写标题、卖点、文案 | 效率低，人力成本占比高       |
| **标准不统一**   | 不同运营人员产出质量参差不齐       | 内容质量波动大，转化率不稳定 |
| **平台规则复杂** | 拼多多/淘宝/小红书各有不同规则     | 易违规，审核通过率低         |
| **迭代周期长**   | 新品上架需要等待内容生产           | 错过最佳上架时机             |

### 解决方案

本系统通过 **Agent 多步骤拆解** + **RAG 规则增强** + **LLM 智能生成**，实现：

- ✅ **效率提升 10x**：从人工1小时/件 → AI 30秒/件
- ✅ **质量标准化**：统一输出规范，符合各平台规则
- ✅ **多平台适配**：一键生成拼多多、淘宝、小红书等多平台内容
- ✅ **全流程闭环**：输入商品信息 → 自动生成 → 历史记录 → 数据分析

## ✨ 项目亮点

### 亮点1：从单次LLM调用升级为Agent多步骤系统

**传统方案**：直接调用LLM生成所有内容

```
用户输入 → LLM → 输出（质量不可控）
```

**本系统方案**：任务拆解 + 步骤执行

```
用户输入 → Agent拆解 → 步骤1标题 → 步骤2卖点 → 步骤3FAQ → 合并输出
```

**优势**：

- ✅ 每步骤独立可控，便于调试和优化
- ✅ 支持步骤级重试，提升成功率
- ✅ 可观测性强，每步骤耗时和Token消耗可追踪

### 亮点2：引入RAG电商规则增强生成稳定性

**问题**：LLM可能生成不符合平台规则的内容

**解决方案**：生成前检索知识库，注入规则

```
用户输入 → RAG检索规则 → 规则增强Prompt → LLM生成
```

**知识库内容**：

- 拼多多标题规则（30字限制、促销词要求）
- 淘宝标题规则（60字限制、关键词布局）
- 小红书风格规则（种草语气、话题标签）

### 亮点3：实现电商内容生成自动化闭环

**完整流程**：

```
商品入库 → AI生成 → 审核发布 → 效果分析 → 迭代优化
     ↑                                      │
     └──────────────────────────────────────┘
```

### 亮点4：前后端 + AI完整系统工程实现

**技术栈**：

| 层级   | 技术                | 职责               |

## 🔧 快速开始

### 环境要求

- Python 3.9+
- Node.js 18+
- MySQL 8.0+

### 后端启动

```bash
cd backend

# 创建虚拟环境
python -m venv venv
venv\Scripts\activate  # Windows

# 安装依赖
pip install -r requirements.txt

# 创建数据库
mysql -u root -p
CREATE DATABASE ai_ecommerce CHARACTER SET utf8mb4;

# 初始化数据
python init_data.py

# 启动后端服务
python main.py
```

后端服务启动在：http://localhost:8000

### 前端启动

```bash
cd frontend

# 安装依赖
npm install

# 启动开发服务器
npm run dev
```

前端服务启动在：http://localhost:5173

## 🎯 面试讲解要点

### 面试官可能问的问题

1. **为什么要用Agent而不是直接调用LLM？**
   - 任务复杂度高，单次调用难以保证质量
   - 步骤拆解便于调试和优化
   - 支持步骤级重试和可观测性

2. **为什么需要RAG？**
   - LLM知识有限，需要注入领域规则
   - 确保生成内容符合平台规范
   - 支持规则动态更新

3. **如何保证结构化输出？**
   - 强制JSON格式要求
   - 输出格式验证
   - 失败重试机制

4. **系统如何扩展？**
   - 新增步骤只需添加Prompt模板和处理逻辑
   - 新增模型只需实现LLMClient接口
   - 新增规则只需添加到知识库
