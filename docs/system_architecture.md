# 系统架构说明

## 1. 设计动机

### 1.1 为什么要这样设计？

在电商内容生成场景中，我们面临以下挑战：

| 挑战             | 描述                                      | 解决方案        |
| ---------------- | ----------------------------------------- | --------------- |
| **任务复杂度高** | 单次生成需要处理标题、卖点、FAQ等多个任务 | Agent多步骤拆解 |
| **规则变化频繁** | 各平台规则不断更新                        | RAG动态规则注入 |
| **输出格式要求** | 前端需要结构化数据                        | 强制JSON输出    |
| **系统可扩展**   | 需要支持新增平台和模型                    | 模块化架构设计  |

### 1.2 设计原则

```
1. 单一职责：每个模块只负责一件事
2. 依赖倒置：高层模块不依赖低层模块，都依赖抽象
3. 开闭原则：对扩展开放，对修改关闭
4. 接口隔离：使用接口而非具体实现
5. 可观测性：每个环节都有执行记录和统计
```

### 2.2 分层职责说明

| 层级           | 职责                         | 技术栈              | 设计原因                         |
| -------------- | ---------------------------- | ------------------- | -------------------------------- |
| **用户输入层** | 提供用户界面，接收商品信息   | Vue3 + Element Plus | 直观的用户交互，快速响应用户操作 |
| **API接口层**  | 接收请求、验证参数、分发业务 | FastAPI             | 高性能、自动文档、类型安全       |
| **核心服务层** | 业务逻辑、AI能力、数据处理   | Python              | 模块化设计，便于扩展和测试       |
| **数据持久层** | 数据存储和检索               | MySQL               | 成熟稳定，事务支持，社区活跃     |
| **外部服务层** | LLM模型调用                  | 阿里云/DeepSeek     | 利用云服务的高可用性和模型能力   |

## 4. 模块职责说明

### 4.1 模块职责表

| 模块                            | 职责              | 关键类/函数                              | 设计原因                             |
| ------------------------------- | ----------------- | ---------------------------------------- | ------------------------------------ |
| **api/**                        | REST API接口层    | `/api/v1/generate`, `/api/v1/history`    | 统一入口，便于前端对接               |
| **services/llm_service.py**     | 统一LLM调用层     | `LLMService.call()`, `call_structured()` | 屏蔽底层差异，支持多模型扩展         |
| **services/agent_service.py**   | Agent任务拆解引擎 | `EcommerceAgent.run()`                   | 将复杂任务拆解为独立步骤，提升可控性 |
| **services/prompt_service.py**  | Prompt模板管理    | `PromptService.render_prompt()`          | 分离Prompt与代码，便于维护和优化     |
| **services/rag_service.py**     | RAG知识检索       | `RAGService.retrieve()`                  | 动态注入领域知识，提升生成质量       |
| **services/history_service.py** | 历史记录管理      | `HistoryService.create_record()`         | 数据持久化，支持查询和分析           |
| **models/**                     | 数据库模型        | SQLAlchemy ORM                           | 类型安全的数据库操作                 |
| **core/**                       | 配置与基础能力    | Settings, Database                       | 集中管理配置，便于环境切换           |

### 6.3 RAG升级路线

| 阶段 | 方案                               | 适用场景                 | 升级原因             |
| ---- | ---------------------------------- | ------------------------ | -------------------- |
| 当前 | Sentence Transformers + 内存向量库 | 小型知识库(<1000条)      | 快速开发，零依赖     |
| 中级 | FAISS本地向量库                    | 中型知识库(1000-10000条) | 性能提升，支持索引   |
| 高级 | Milvus/Pinecone                    | 大型知识库(>10000条)     | 分布式部署，水平扩展 |

## 8. 安全设计

### 8.1 安全措施

| 安全措施        | 实现方式                           | 目的             |
| --------------- | ---------------------------------- | ---------------- |
| **API密钥管理** | 使用环境变量存储，不在代码中硬编码 | 防止密钥泄露     |
| **请求验证**    | 使用Pydantic进行参数验证           | 防止恶意输入     |
| **CORS配置**    | 配置允许的源地址                   | 防止跨站请求伪造 |
| **数据加密**    | 数据库密码使用环境变量             | 保护敏感信息     |
| **请求限制**    | 可配置请求频率限制                 | 防止API滥用      |

### 8.2 安全最佳实践

```
1. 不在前端暴露任何API密钥
2. 不在日志中记录完整的Prompt内容
3. 对用户输入进行严格验证
4. 定期更新依赖库，修复安全漏洞
5. 使用HTTPS传输数据
```

## 9. 面试亮点

### 9.1 为什么这样分层设计？

```
1. 关注点分离：每层只负责自己的职责，便于理解和维护
2. 可扩展性：新增功能只需修改对应层级，不影响其他层级
3. 可测试性：每层可以独立测试，降低测试复杂度
4. 技术选型灵活：每层可以选择最适合的技术，不受其他层限制
```

### 9.2 系统如何保证高可用性？

```
1. LLM服务多Provider支持：一个Provider故障时可切换到其他Provider
2. 步骤级重试：单步骤失败可重试，不影响整体流程
3. 数据库事务：确保数据一致性
4. 日志记录：便于问题定位和故障恢复
```

### 9.3 如果用户量扩大，如何扩展？

```
1. 前端：CDN加速，静态资源分离
2. API层：负载均衡，多实例部署
3. Agent层：异步任务队列，并行处理
4. 数据库：读写分离，分库分表
5. LLM调用：请求缓存，模型路由
```

### 9.4 系统的薄弱环节是什么？

```
1. LLM API调用延迟：网络延迟和模型响应时间不确定
2. RAG检索准确性：向量检索可能返回不相关的知识
3. JSON解析失败：LLM可能生成不符合格式的输出
4. 知识库更新：规则变更需要手动更新
```
